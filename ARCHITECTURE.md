\# FinTech Credit System Architecture



\## Выбранный подход: Event-Driven Architecture (EDA) с элементами Saga



\### Сравнительный анализ архитектур



| Критерий | Монолит (Этап 1) | Синхронные микросервисы (Этап 2) | EDA + Saga (Этап 3-4) |

|----------|------------------|--------------------------------|----------------------|

| \*\*Отказоустойчивость\*\* | Падение любого компонента → весь сервис недоступен | Падение Scoring → Gateway возвращает 500, заявка теряется | Заявки накапливаются в очереди, обрабатываются позже |

| \*\*Масштабирование\*\* | Только вертикальное | Можно масштабировать отдельно, но синхронные вызовы создают блокировки | Каждый сервис масштабируется независимо |

| \*\*Время ответа\*\* | Линейно растёт с нагрузкой | Зависит от самого медленного сервиса | Клиент получает мгновенный ответ (асинхронно) |

| \*\*Консистентность\*\* | ACID транзакции (сильная) | Сложно обеспечить | Eventual consistency |

| \*\*Аудит\*\* | В одной БД — легко | Нужно агрегировать логи | Correlation ID позволяет отследить всю цепочку |



\### Почему EDA подходит для FinTech?



1\. \*\*Отказоустойчивость\*\* — при падении Scoring Service заявки не теряются, а ждут в очереди

2\. \*\*Пиковые нагрузки\*\* — брокер сообщений работает как буфер

3\. \*\*Независимость команд\*\* — сервисы могут развиваться независимо

4\. \*\*Audit trail\*\* — все события сохраняются



\### Когда eventual consistency критична?



\*\*Критично (нельзя использовать EDA):\*\*

\- Списание/зачисление денег (double spending)

\- Обновление баланса счета

\- Блокировка средств



\*\*Некритично (наш случай):\*\*

\- Статус кредитной заявки ("processing" -> "approved")

\- Обновление клиентского профиля

\- Отправка уведомлений



\### Saga (Choreography)



В нашей реализации:

1\. Gateway публикует `ApplicationCreated`

2\. Scoring публикует `ScoreCalculated`

3\. Deal подписывается и создаёт договор

4\. Если Deal не может создать договор → публикует `DealCreationFailed`

5\. Saga Compensator откатывает статус заявки



\### Correlation ID



Каждое событие содержит `application\_id`, который передаётся через все сервисы. Это позволяет:

\- Склеить логи в единую цепочку

\- Отследить путь заявки через систему

\- Диагностировать проблемы



\### API Gateway: "Умный" vs "Тупой"



| Тип | Назначение | Когда антипаттерн |

|-----|-----------|------------------|

| \*\*Тупой (Reverse Proxy)\*\* | Маршрутизация, SSL termination, rate limiting | — |

| \*\*Умный (API Composition)\*\* | Агрегация данных, трансформация протоколов | Когда в Gateway начинает оседать бизнес-логика → Distributed Monolith |



В нашей реализации Gateway — "умный" (агрегирует данные для `/dashboard`), но бизнес-логика (скоринг, создание договора) остаётся в сервисах.



\### Результаты нагрузочного тестирования



| Этап | Поведение при отказе Scoring |

|------|------------------------------|

| Этап 1 (монолит) | Полный отказ системы |

| Этап 2 (синхронные) | 500 ошибка, потеря заявки |

| Этап 3-4 (EDA) | Заявки в очереди, 0 потерь |



\### Вывод



Для FinTech системы с высокими требованиями к отказоустойчивости и пиковым нагрузкам Event-Driven Architecture с Saga является оптимальным выбором. Ценой становится eventual consistency, которая в данном контексте (кредитные заявки) допустима.
